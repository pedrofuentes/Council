# Council TUI — Milestone 9.6 (M5 Panel Authoring) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Author panels inside the TUI — create a panel from a multi-select of experts, edit its members, delete it (with retained-session warnings), and auto-compose a panel from a topic (engine-backed) with a preview + confirm. Behind `COUNCIL_TUI=1`; the full CLI stays untouched.

**Architecture:** A pure, fully-tested `panel-authoring` adapter wraps create/setMembers/delete over an injected `PanelLibraryRepository` + filesystem, mirroring the CLI's `persistPanelArtifacts` (write DB row → YAML `<dataHome>/panels/<name>.yaml` with `wx` → `setMembers` → docs dir; rollback on failure) and the CLI's delete flow (unlink YAML + `rm -rf` panel dir → DB delete; debate sessions preserved). A new reusable **`MultiSelectList`** component (the codebase has only single-select today) drives member selection. Create/edit/delete screens reuse the M3 input-capture + mode-gated pattern and the M4 destructive-action gating. Auto-compose (the only engine-dependent step) calls `autoComposePanel(topic, engine, opts)` and is unit-tested with `MockEngine` (which has a deterministic compose fallback).

**Tech Stack:** TypeScript ESM, Ink 7, React 19, `ink-text-input`, `react-router`, Zod (`PanelDefinitionSchema`), `yaml`, Vitest + `ink-testing-library` + `MockEngine`.
