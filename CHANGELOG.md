# Changelog — Council

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/),
and this project adheres to [Semantic Versioning](https://semver.org/).

## [Unreleased]

### Changed

- **`council panels` renamed to `council sessions`** — the command that lists debate-session records from the local DB is now `council sessions`. The previous name was confusingly close to `council panel` (which manages the panel YAML library: `create | list | inspect | edit | docs`). The DB table is still named `panels` — only the CLI surface changed. File rename: `src/cli/commands/panels.ts` → `src/cli/commands/sessions.ts`; export rename: `buildPanelsCommand()` → `buildSessionsCommand()`.
- Profile analyzer retries on stream errors with configurable `timeoutMs` (#427).
- Profile analyzer exposes `onWarning` callback for cleanup-failure reporting (#427).

### Fixed

- **`council panel docs unlink` race with concurrent scanner** — the panel-document scanner now atomically re-checks linked-folder membership immediately before writing FTS and `panel_documents` rows for each file, wrapping the re-check + writes in a `BEGIN IMMEDIATE` transaction. Previously, an unlink that committed during a scan could be silently undone because the scanner had already snapshotted folder membership at scan-start and never re-checked before its per-file writes (#424).
- Copilot adapter now unregisters SDK event listeners after each `send()`, preventing memory leak in long chat sessions (#491).
- **Ctrl+C during `@convene` debate**— interrupting an inline structured debate via Ctrl+C now aborts the debate cleanly, persists any partial expert response that was already streamed, and surfaces a visible warning if that flush itself fails. The deferred `@convene` user turn is only committed after the partial expert turn writes successfully so an aborted, never-answered prompt no longer leaves an orphan user row in chat history (#466).
- **Graceful CLI error handling** — user-facing errors (not-found, validation, missing flags) now print a clean one-line message to stderr and exit with code 1 instead of dumping full Node.js stack traces. Internally, errors that have already been written to stderr by command handlers throw `CliUserError` (suppressed by the top-level handler); all other errors have their message printed without a stack trace.
- Persona profile prompts now render `epistemicStance` field correctly instead of silently dropping it (#407).
- Corrupt profile JSON in the database now emits a warning and degrades gracefully instead of crashing (#408).

### Added

- Graceful Ctrl+C handling during `council chat` — sends interrupt signal instead of killing the process (#465).
- Fallback to DB-cached expert metadata when expert YAML file is deleted between chat sessions (#465).
- **Model selection**— experts and panels can now specify a preferred model. Use `--model` on `council expert create` or `council panel create`, or set `defaults.model` in panel YAML. The 3-layer resolution hierarchy is: expert model → panel default → global config (`config.yaml` `defaults.model`). Use `council doctor --models` to list known models and `--online` to verify accessibility.

- **Chat session infrastructure** (Roadmap 5.1) — new `chat_sessions` and `chat_turns` tables (migration 005) provide the persistence substrate for open-ended conversations alongside the structured-debate `debates`/`turns` tables (per ADR-010). `ChatRepository` (`src/memory/repositories/chat-repository.ts`) exposes `createSession`, `findSessionById`, `findActiveSession`, `listSessions`, `archiveSession`, `updateSummary`, `addTurn`, `getTurns`, `getTurnCount`, `getLatestSeq`, and `searchTurns`. `addTurn` allocates per-session monotonic `seq` atomically via `INSERT … SELECT COALESCE(MAX(seq), 0) + 1 …` so concurrent appends cannot collide on the `UNIQUE (chat_id, seq)` constraint declared in migration 005. Domain types `ChatSession` and `ChatTurn` — with `ChatRole = "user" | "expert"` — live in `src/core/chat/chat-session.ts` and are the shared currency between the chat command, renderer, and context manager.

- **1:1 expert chat** (Roadmap 5.2) — `council chat <expert-slug>` opens a persistent conversational REPL with an expert from the library. By default the active chat session for that expert is **resumed** (transcript replayed, prior turns visible to the engine on the first new prompt); `--new` archives the active session and starts fresh; `--list` prints every chat conversation across all targets (active + archived) with target, timestamps, and turn counts; `--history` (boolean) shows the archived sessions for a given target, read-only, without registering the expert. The interactive loop is a Node `readline` REPL (`You >` prompt) that persists each user turn before sending it to the engine and persists the streamed expert response on `turn.end`. Engine failures during a turn are non-fatal: the error is surfaced inline via the chat renderer's system-message channel and the loop continues so the prompt can be retried. Implementation: `src/cli/commands/chat.ts` (`runExpertChat`).

- **Rolling-summary context manager** (Roadmap 5.3) — `src/core/chat/context-manager.ts` keeps the most-recent N turns of a chat verbatim (configurable via `recentTurnCount`) and compresses everything older into a single LLM-generated rolling summary persisted on `chat_sessions.summary` / `summary_through_seq`. The `ContextManager` interface exposes `getContext(chatId)` returning `{ summary, recentTurns }`, `maybeSummarize(chatId)` (run after appending turns; returns `true` iff a new summary was produced), and `forceSummarize(chatId)` (resolves silently on empty input or LLM failure). The summarizer is invoked on a transient "Context Summarizer" expert that the module tears down via best-effort `removeExpert` (cleanup errors are silently swallowed in `runSummarizer()`, matching the module-wide best-effort contract: `forceSummarize` resolves silently on LLM failure and every failure path in `maybeSummarize` is best-effort). Untrusted prior-summary and transcript fields are wrapped in `<prior_summary>` / `<transcript>` fences with every `<` and `>` escaped to `&lt;`/`&gt;` so a hostile turn cannot break out of the data fence. **Wired into `council chat`**: both 1:1 (`runInteractiveLoop`) and panel (`runPanelInteractiveLoop`) paths now source recent turns + persisted summary via `contextMgr.getContext(chatId)` (replacing the manual `getTurns(afterSeq:…)` slicing) and call `maybeSummarize` after each persisted expert turn so summaries roll forward automatically. Loading-context, summarization, and retrieval failures are wrapped in best-effort `safeGetContext` / `safeMaybeSummarize` / `safeRetrieveSnippets` helpers that surface a single sanitized warning and let the chat continue.

- **RAG retrieval in chat** (Roadmap 6.3 / TSD §7) — both 1:1 and panel `council chat` now call the FTS5 document retriever (`src/core/documents/retriever.ts`) on every user turn and prepend matching snippets to the user message before prompting the expert. 1:1 chat scopes retrieval by `expertSlug`; panel chat scopes by `panelName`. Snippets are appended via the new exported `appendReferenceDocuments(userMessage, snippets)` helper, which fences each snippet between `<<<DOC source="…">>>` / `<<<END>>>` delimiters and prefixes the block with an explicit instruction telling the model to treat the fenced content as untrusted reference data — never as instructions or role changes — even if the snippet text appears to ask for action. Forged fence markers and stray newlines inside snippet source/content are neutralized so a malicious indexed document cannot break out of the data fence. Retrieval failures degrade gracefully: a single sanitized warning is surfaced via the renderer and the chat turn proceeds with no reference block.

- **Chat renderer** (Roadmap 5.7) — `src/cli/renderers/chat-renderer.ts` is the chat-specific output surface (separate from the debate renderers in `src/cli/renderers/`). Per-expert color is assigned deterministically from an 8-color chalk palette by registration order so a given expert always renders in the same color across resumes and across panel/1:1 chats; the assignment wraps so panels with more experts than colors stay consistent. Streaming chunks are written incrementally as they arrive; the prompt is `You >`; system messages are tagged `info` / `warn` / `error` with distinct prefixes. Defense-in-depth sanitisation runs on every rendered surface: ANSI CSI/OSC sequences and C0 control bytes are stripped, lone carriage returns are removed (so a malicious `\r` cannot rewrite the current line), and Unicode line separators (NEL / U+2028 / U+2029) are collapsed to a space on single-line surfaces (prompts, headers) so they cannot break the prompt frame.

- **Panel CLI commands** (Roadmap 4.4) — `council panel create | list | inspect | edit` mirrors the expert library CLI for panel definitions. `create` is an interactive readline wizard that picks experts from the library (numeric multi-select with validation), prompts for `name`, `description`, `mode`, and writes `~/Council/panels/<name>.yaml` plus a `panel_library` row with `panel_members` joins. `list` (`--format table|json`) reads from `panel_library`. `inspect <name>` shows panel metadata and resolved expert roster. `edit <name>` opens `$VISUAL`/`$EDITOR` on the panel YAML, re-validates with `PanelDefinitionSchema`, refuses name renames, and re-syncs the DB row + checksum. The CLI is backed by `PanelLibraryRepository` (`src/memory/repositories/panel-library-repo.ts`) which owns CRUD over `panel_library` + `panel_members` (composition is normalised, not stored as JSON). Implementation: `src/cli/commands/panel.ts`.

- **Memory model enforcement for generic experts** (Roadmap 7.1) — `buildSystemPrompt()` in `src/core/prompt-builder.ts` now hard-guards the `[N] PERSONA PROFILE` section on `def.kind === "persona"`: a profile passed in for a generic (`def.kind !== "persona"`) expert is silently ignored and section numbering shifts back to the canonical 8 sections. Generic experts thus receive only `[7] MEMORY` (debate-derived recall); persona experts can additionally receive `[N] PERSONA PROFILE`. The kind-based guard prevents accidental cross-contamination — e.g. a generic "VC Partner" expert that happened to have a stale `persona_profiles` row from an earlier persona configuration cannot pull that profile into its prompt.

- **Memory / persona-profile separation in `council memory reset`** (Roadmap 7.4) — `council memory reset` clears debate transcripts, turns, and the `extracted_memory_json` cache on every expert in the panel, but **does not** touch the `persona_profiles` table. Persona profiles are document-derived (LLM distillation of `~/Council/experts/<slug>/docs/`) and represent stable identity, not deliberation history; users can therefore wipe debate state without losing the work invested in document indexing. The only way to clear or rebuild a persona profile is `council expert train --retrain <slug>`, which re-runs the analyzer against the current docs folder. The split is documented in the `memory reset` confirmation output so users know what will and will not be cleared.

- **Cross-panel awareness in 1:1 chat** (Roadmap 7.2 + 7.3)— when you `council chat <persona-slug>`, the expert's system prompt now includes a `[PANEL MEMBERSHIPS]` section listing the other panels they belong to (with co-members and panel descriptions), so 1:1 conversations are informed by the shared context across panels. Memberships are ordered most-recently-active first (by `panel_library.updated_at`) and capped at 5 entries to bound prompt size. All externally-sourced fields (panel names, descriptions, co-member display names) are run through the same sanitiser used for persona profile fields — C0 controls stripped, line breaks collapsed, `[N]` section markers defanged to `(sec-N)`, length capped at 2000 chars — so a panel YAML cannot forge section headers in the privileged system prompt. The query is best-effort: a DB failure surfaces a one-line warning (not silent) and chat continues without the section. Panel chat is unchanged. Implementation: `getExpertPanelMemberships()` in `src/core/panel-membership-query.ts`, `renderPanelMemberships()` and `PanelMembership` in `src/core/prompt-builder.ts`.

- **Panel document folder** (Roadmap 6.7) — panels now have a shared reference-document corpus indexed into FTS5 alongside expert documents. Three new CLI subcommands:
  - `council panel docs <name>` — list managed and externally-linked folders for a panel and their document counts.
  - `council panel docs link <name> --path <folder>` — link an external folder so its documents are scanned and indexed at chat start.
  - `council panel docs unlink <name> --path <folder>` — remove a linked folder and clean up its `panel_documents` rows + FTS index entries. If FTS cleanup fails the unlink aborts with a non-zero exit and the linked-folder + `panel_documents` rows are preserved so the operation can be retried.

  `council panel create` auto-provisions `~/Council/panels/<name>/docs/` so users can drop shared docs without explicit linking. On `council chat <panel>`, the panel scanner walks both the managed and linked folders, detects new/changed files via SHA-256 checksums, extracts and indexes them under `source_type='panel'`, prunes any tracked documents that have disappeared from disk (both `panel_documents` rows are marked `removed` and the FTS `document_index` entries are deleted; transient folder-level failures do NOT trigger pruning, to avoid wiping the index on an unmounted drive), and surfaces a one-line summary plus a warning if any folder failed to scan (distinguishing the managed folder from linked folders). If every discovered document fails to process, a `⚠ All N documents failed to process. Check file formats and permissions.` warning is surfaced so the user is not left with a silent empty corpus. **Security:** linked folders are canonicalised once and passed as `confinementRoot` to BOTH the detector and the extractor (defense-in-depth against detector→extractor TOCTOU symlink swaps); `panel docs link` uses `fs.lstat` and rejects symlinked paths up-front so the link command and the scanner agree on the confinement policy. New schema in migration 009: `panel_linked_folders` and `panel_documents` (FK CASCADE from `panel_library`).

- **Recency weighting for persona document analysis** (Roadmap 6.8) — `analyzeDocuments()` (`src/core/documents/profile-analyzer.ts`) now annotates each document block in the analyzer meta-prompt with a recency-weight tag (`[Weight: 0.NN]`) computed from the document's `modifiedAt` timestamp via exponential decay: `weight = 2^(-ageDays / halfLifeDays)` (so age = 0 → 1.0, age = halfLife → 0.5, age = 2× halfLife → 0.25). The meta-prompt explicitly instructs the LLM to weight more-recent material more heavily when distilling the persona profile. Half-life is configurable via `AnalyzeOptions.recencyWeightHalfLife` (`DocumentProcessor` default: 90 days). Edge cases: future-dated documents and `halfLifeDays <= 0` clamp to weight 1.0 so a misconfigured value cannot zero out every document; documents without a `modifiedAt` render without a weight tag (back-compat). New helper: `calculateRecencyWeight(documentDate, now, halfLifeDays)`. The `now` reference timestamp is injectable via `AnalyzeOptions.now` for deterministic tests. NOTE: the analyzer preserves the caller's input order — it does not itself sort documents by `modifiedAt`. Weight tags reflect age regardless of input order.

- **Persona profile analysis** (Roadmap 6.2) — new `analyzeDocuments()` in `src/core/documents/profile-analyzer.ts` distills a set of persona-expert documents into a structured `PersonaProfile` (`{ communicationStyle, decisionPatterns, biases, vocabulary, epistemicStance, lastUpdated, documentCount, totalWords }`) by registering a transient "Profile Analyzer" expert with the engine, sending a meta-prompt containing the document contents, and parsing the JSON response. The transient analyzer is always torn down in a `finally` block (engine cleanup failures surface a one-line warning rather than masking the analysis result). Parsing is permissive: a leading ```` ```json ```` / ```` ``` ```` fence is stripped from the response, then `JSON.parse` runs; if the result lacks a non-empty `communicationStyle` or `epistemicStance` it is treated as malformed and a single retry is issued. Persistent failure throws `Error("Profile analyzer returned unparsable JSON after retry")`. The meta-prompt explicitly marks document content as untrusted data (mirroring `src/memory/memory-extractor.ts`); every `<` in interpolated content is escaped to `&lt;` so an XML-like closing tag cannot break the `<documents>` fence, and `existingProfile` fields are run through `sanitizePromptField` (C0 controls stripped, line breaks collapsed, `[N]` section markers defanged to `(sec-N)`, length capped at 2000 chars). Profiles are persisted to the new `persona_profiles` table (migration 008) keyed by `expert_slug` and survive `council memory reset` (preserved per Roadmap 7.4 — only debate transcripts and extracted memory are cleared). The prompt-builder injects the profile into a new `[N] PERSONA PROFILE` section (`src/core/prompt-builder.ts:renderPersonaProfile`); the section is suppressed when `def.kind !== "persona"` (Roadmap 7.1).

- **`buildSystemPrompt(def, memory, task, personaProfile?, panelMemberships?)`** — `src/core/prompt-builder.ts` extended to accept two optional inputs that shift section numbering dynamically. Without either, the prompt has the canonical 8 sections (`[1] IDENTITY` … `[8] CURRENT TASK`). With `personaProfile` AND `def.kind === "persona"`, a `PERSONA PROFILE` section is injected after `[7] MEMORY` and `CURRENT TASK` shifts to `[9]`. With `panelMemberships`, a `PANEL MEMBERSHIPS` section is injected before `CURRENT TASK`. Both arguments are optional and back-compat: existing 3-arg call sites are unchanged.

- **Document tracking for persona experts** (Roadmap 6.1) — new `expert_documents` table (migration 006) tracks individual files in `~/Council/experts/<slug>/docs/` with `{ id (ULID), expert_slug (FK CASCADE → expert_library.slug), file_path, filename, checksum (SHA-256), size_bytes, word_count, status, processed_at, created_at }`, with a `UNIQUE (expert_slug, file_path)` constraint and `idx_expert_documents_slug` covering `(expert_slug, status)` for the change-detection hot path. Status transitions: `pending → processed | failed`; deletions reconciled via `removed`. Combined with the FTS5 `document_index` (migration 007) this is the substrate for incremental indexing, on-demand reprocessing (6.4), and recency weighting (6.8). New repository: `DocumentRepository` (`src/memory/repositories/document-repository.ts`).

- **On-demand persona document processing** (Roadmap 6.4) — when you `council chat <persona-slug>`, Council now auto-detects new, modified, or deleted files in `~/Council/experts/<slug>/docs/`, extracts and indexes new/changed files, reconciles deletions (the FTS index is pruned and the DB row is marked removed), refreshes the persona profile, and only then registers the expert with the engine — so subsequent turns immediately benefit from the latest reference material. Idempotent (unchanged docs are skipped via SHA-256 checksum), incremental (only new/changed files are re-extracted), and non-blocking (per-file failures are reported as warnings without aborting the chat; analyzer failures surface a one-line warning and the previous profile is preserved). Empty docs folders surface a one-line info message and the persona runs as a generic expert. Generic (non-persona) experts are unaffected. **Security:** the docs root must be a real directory — symlinks/junctions as the root itself are rejected up front, and per-file confinement uses a once-resolved canonical root passed through the detector and extractor (closing root-swap TOCTOU windows). Implementation: `createDocumentProcessor` in `src/core/documents/processor.ts`; integration in `runExpertChat` in `src/cli/commands/chat.ts`.
- **TOCTOU-hardened document extractor** — `extractDocument` now performs fd-based reads (open → fd-stat → realpath → lstat compare → confinement check → read via handle) so a symlink swap between path resolution and read cannot exfiltrate data outside the configured `confinementRoot`. Back-compat preserved when `confinementRoot` is omitted.

- **@mention routing in panel chat** (Roadmap 5.5) — type `@<expert-slug> <message>` to direct a question to one or more specific panelists; only the named experts respond, with each persisted chat turn carrying an `isMention: true` flag. Multiple mentions are supported (`@cto @sre …`) and experts respond in the panel's declared order regardless of mention order. Unknown slugs surface a one-line error listing the available panel slugs and the malformed user turn is NOT persisted, so the prompt can be retyped cleanly. Non-mentioned panelists still see the exchange in their context on subsequent turns. The parser is bypassed in 1:1 expert chat (a literal `@x …` is processed as plain text). Implementation: `parseUserInput()` in `src/core/chat/mention-parser.ts`.

- **`@convene <topic>` — inline structured debate** (Roadmap 5.6) — within an active panel chat, type `@convene <topic>` to interrupt the freeform conversation with a structured 4-phase deliberation (opening → cross-examination → rebuttal → synthesis; cross-exam skipped for single-expert panels) using the same engine and registered experts. Each debate `turn.end` is persisted as a regular `ChatTurn` so the deliberation joins the chat history and is visible to all subsequent turns. After the debate completes (or completes-with-partial-failures), control returns to freeform chat with a confirmation or interruption notice. Per-turn engine errors are non-terminal; partial results are preserved on interruption.

- **Template migration to the library format** (Roadmap 4.6) — on the first `council convene --template <name>` run after upgrade, Council automatically extracts the inline experts from each built-in panel (`architecture-review`, `career-coaching`, `code-review`, `incident-postmortem`, `startup-validation`) into standalone YAMLs under `~/Council/experts/<slug>.yaml` and rewrites each panel into `~/Council/panels/<name>.yaml` referencing those experts by slug. Migration is **non-destructive** (it never overwrites an existing user expert or panel YAML), **idempotent** (safe to re-run; short-circuits once both the filesystem and the `expert_library` AND `panel_library` SQLite tables show migrated state), and **DB-reset-resilient** (if the on-disk YAMLs survive a DB wipe of either the expert or panel tables, the next run re-registers library rows from disk, refreshes stale `panel_library` metadata from user-edited panel YAML, and materialises any inline expert overrides into `expert_library` so `panel_members` FK constraints are satisfied — all without rewriting files). Slug collisions across panels are resolved by reusing identical experts and suffixing genuinely-different ones with the panel name (e.g. `sre-incident-postmortem`). Programmatic entry point: `migrateBuiltInTemplates(dataHome, library, db, options?)` and `isMigrationNeeded(dataHome, db?)` in `src/core/template-migration.ts`.

- **Panel composition model** (Roadmap 4.2) — `PanelDefinitionSchema` now accepts either a slug string (referencing the expert library from P4-T2) or an inline `ExpertDefinition` for each `experts` entry, so a single panel YAML can mix reusable library experts with one-off inline definitions. New loader functions:
  - **`loadUserPanel(name, dataHome)`** — reads a user-authored YAML from `<dataHome>/panels/<name>.{yaml,yml}` (path-traversal-guarded; throws a typed `PanelNotFoundError` when absent so callers can fall back without parsing error strings).
  - **`listUserPanels(dataHome)`** — returns user panel names without extension, sorted; empty array on a fresh install.
  - **`loadPanel(name, dataHome)`** — user panels in `<dataHome>/panels/` take precedence over the built-in templates that ship with Council, so a user can override a stock panel by name without touching the package.
  - **`resolveExperts(entries, library)`** — looks slug refs up in an `ExpertLibrary`, passes inline definitions through unchanged, and reports any missing slugs in a single collection so callers surface one useful error.
  - **`council convene --template <name>`** now resolves slug references against the expert library and reports unresolved slugs (with `'council experts create'` guidance) in a single clear error. Inline-only panels keep the existing fast path. Auto-composed panels reject slug strings from the composer (a non-inline response now aborts auto-compose with a descriptive error rather than silently producing an empty roster).
- **Expert library management CLI** (ROADMAP §4.3) — `council expert create | list | inspect | edit | delete` lets users author, browse, and curate the expert library at `~/Council/experts/*.yaml` (with parallel metadata in the `expert_library` SQLite table). `create` is an interactive readline wizard with a complete set of flags (`--slug`, `--name`, `--role`, `--expertise`, `--stance`, `--personality`, `--persona`, `--persona-description`) for non-TTY use; the `--persona` flag creates a `~/Council/experts/<slug>/docs/` folder for relationship documents. `edit` opens `$VISUAL`/`$EDITOR` (falls back to `vi`/`notepad`), then re-parses the file, refuses slug renames, and syncs the DB metadata row + checksum so library reads stay consistent with disk. `delete` refuses when the expert is a member of any panel unless `--force` is passed. `list` supports `--format table` (default) and `--format json`.


- **LLM-based ExpertMemory extraction** (ROADMAP §3.1) — after each debate, an LLM extraction pass distils each AI expert's prior turns (bounded by the most-recent 5 debates × 20 turns) into structured `{ positions, updatedPriors, unresolved }` and caches the result on the expert row (new `experts.extracted_memory_json` column, migration 003). `recallMemory()` now prefers the cached LLM-distilled memory over the heuristic sentence scan, falling back to the heuristic when the cache is null or all-empty. Cost ceiling: one LLM call per AI expert per debate. Prompt-injection hardened: every `<` in interpolated turn content is escaped to `&lt;` before fencing (same pattern as the LLM summariser).
- **`council convene --heuristic-memory`** and **`council resume --continue --heuristic-memory`** — opt-out flag that skips the post-debate LLM extraction pass and keeps the legacy heuristic-only recall behaviour. Useful for offline tests and air-gapped environments.

- **Context window management** (ROADMAP §2.6, closes #229-ish; see PR #249) — keeps long debates within model limits without sacrificing fidelity. Three knobs, all opt-in:
  - **`council convene --context-scope <scope>`** — `all` (default, unchanged), `same-round` (each expert only sees turns from the current round, including intra-round turns that completed just before), or `recent` (a sliding window of the most-recent N turns across all rounds, default N=10). Backed by `filterPriorTurns()` in `src/core/context/visibility.ts`.
  - **`council convene --summarize-after <n>`** — once round `n` starts, the orchestrator prepends a compact heuristic rolling summary (each expert's first-sentence position + a disagreement-signal flag) to every prompt so the strategy can reference earlier deliberation without every verbatim turn. Backed by `buildRollingSummary()` in `src/core/context/summarizer.ts`.
  - **`DebateConfig.contextConfig.maxPromptChars`** (engine-level; no CLI flag yet) — hard cap on verbatim prior-turn content. When the cap would be exceeded, oldest turns are dropped first; the most recent turn is always retained. Truncation is **opt-in** — omitting `contextConfig` (or omitting `maxPromptChars`) preserves the legacy passthrough behaviour with no implicit cap.
  - Under any `contextConfig` the orchestrator re-plans the strategy _per turn_ so `same-round` and `recent` policies see turns that completed earlier in the same round. Without `contextConfig` the strategy is still planned once per round (legacy behaviour).
- **Rich Ink Terminal UI** (ROADMAP §3.4) — React + Ink-based renderer that auto-activates when stdout is a TTY. Each expert is assigned a stable color from a 6-color palette (cyan, yellow, magenta, green, blue, red) so the same expert renders in the same color across every round. Components include a `PanelRoster` (assembled experts), `RoundHeader` (round separator), `ExpertCard` (colored name header), `StreamingText` (live turn body with caret cursor while writing), `RetryIndicator` (`ink-spinner` + reason while a turn is being retried), `CostIndicator` (premium-request budget), and a completion banner on `debate.end`. The selector lives in `src/cli/renderers/select.ts` and exposes a new `--format auto` value (the new default). Explicit `--format json` and `--format plain` still always override TTY detection so piped output stays deterministic. Adds `ink`, `ink-spinner`, and `react` as runtime dependencies.
- **`council convene --strategy <name>`** and **`council resume --continue --strategy <name>`** (ROADMAP §2.3, closes #212) — wires `ModeratorStrategy` into the freeform debate orchestrator. Supported values: `round-robin` (default), `devils-advocate`, `consensus-check`. The `devils-advocate` strategy accepts an optional `:<expert-slug>` suffix to pin the contrarian (e.g. `--strategy devils-advocate:senior`); when omitted, the first expert is the advocate. The strategy controls per-round turn order _and_ per-turn prompts: round 0 gets the strategy's framing, round 1+ receives a `Prior discussion:` block built from earlier turn content. `consensus-check` can also terminate a debate early via `shouldContinue()` → `debate.end { reason: "consensus" }`. Strategies are ignored in `--mode structured` (phase-prompts continue to drive that mode). The chosen strategy name is persisted to `debates.moderator`.
- **`DebateConfig.strategy?: ModeratorStrategy`** — pluggable hook on the `Debate` orchestrator's freeform path. Defaults to `createRoundRobinStrategy()` so existing callers see the same structural event sequence.
- **`council conclude [panel] --engine <kind> [--format json|plain]`** (ROADMAP §2.7) — synthesizes the latest debate of a panel into a structured decision framework (consensus, tensions, decision matrix, recommendation, confidence). Reads transcript from the local DB, runs a single moderator-style synthesis prompt through a transient "synthesizer" expert, parses the JSON response (tolerates `json` fences and prose-wrapped JSON), validates against a Zod schema, and renders plain or JSON. Defaults to the most recently-created panel when no name is given. Transcript content is delimited in `<transcript>` tags and capped at 50 turns / 50,000 chars to mitigate prompt injection and unbounded context. Synthesis call uses a 60s `AbortSignal.timeout` so a hung engine fails fast. Warnings are emitted when the latest debate isn't `completed` or when the transcript is truncated.

### Hardening

This release lands a wave of defensive fixes from the post-Phase-7 Sentinel sweep. None changes user-facing behavior on the happy path; all close gaps that previously could leave Council in a half-consistent state, leak data across confinement boundaries, or surface generic errors instead of actionable guidance.

- **Atomic operations for non-atomic DB paths (#382, #383, #356, #363, #387, #388)** — six write paths that previously issued multiple statements without a surrounding transaction (panel deletion + cascade cleanup, expert deletion + panel-member detach, document indexer FTS sync, persona profile upsert + checksum update, panel docs unlink, chat session archive) are now wrapped in `BEGIN`/`COMMIT`/`ROLLBACK` on the libsql connection. A failure mid-sequence rolls back cleanly instead of leaving orphan rows or stale FTS entries; user-facing confirmation lines fire only after a successful `COMMIT`.
- **Profile analyzer sanitization (#358, #366)** — `existingProfile` fields injected into the analyzer meta-prompt now flow through the shared `sanitizePromptField` (extracted to `src/core/prompt-sanitize.ts`) before the fence-specific `<` → `&lt;` escape: C0 control bytes stripped, Unicode line breaks (NEL / U+2028 / U+2029) and CR/LF runs collapsed, bracketed `[N]` section markers defanged to `(sec-N)`, length capped at 2000 chars. Prevents a hostile persisted profile field from forging section headers in the analyzer prompt.
- **Panel document security — confinement, stale-file pruning, symlink consistency (#385, #386, #389, #390)** — linked-folder paths are canonicalised once and passed as `confinementRoot` to BOTH the detector and the extractor (defense-in-depth against detector→extractor TOCTOU symlink swaps); `panel docs link` uses `fs.lstat` and rejects symlinked paths up front so the link command and the scanner agree on confinement policy. Stale `panel_documents` rows whose files have disappeared from disk are pruned with their FTS entries on every scan, but transient folder-level failures (unmounted drive, permission flicker) explicitly do NOT trigger pruning so a temporary outage cannot wipe the index.
- **Memory recall for persona experts (#364)** — `recallMemory()` previously hardcoded the `[7]` section index when matching persisted system prompts, so when a persona expert had a `[N] PERSONA PROFILE` section that shifted `MEMORY` to `[8]`, the recall patch silently no-op'd. Section numbering is now derived from the actual prompt structure (the patch path locates the section header via `lastIndexOf` rather than a fixed offset) so persona experts get their recalled memory injected correctly.
- **Error handling improvements — timeouts, stream error rejection, transaction-wrapped reset (#359, #360, #361, #341, #342, #403)** — engine `send()` calls in `conclude` (manual `AbortController` + `setTimeout` in `src/cli/commands/conclude.ts`) and the profile analyzer (`AbortSignal.timeout` in `src/core/documents/profile-analyzer.ts`) now bound the wait so a hung backend fails fast with an actionable message instead of hanging indefinitely; stream `error` events are surfaced as rejected promises rather than silently swallowed; and `council memory reset` (default mode) is wrapped in a single `BEGIN`/`COMMIT`/`ROLLBACK` so a failure between the debate-delete and expert-memory-clear loops cannot leave the panel half-reset.

### Fixed

- **`council memory reset` (default mode) is now atomic (#403)** — the default reset path previously deleted every debate and then cleared each expert's `extractedMemoryJson` as independent statements with no surrounding transaction, so a failure between or during either loop left the panel half-reset (debates gone but expert memory preserved, or vice versa) with no recovery. Both loops are now wrapped in a single `BEGIN`/`COMMIT`/`ROLLBACK` on the libsql connection (matching the convention used by `document-repository.clearForRetrain` and the documents indexer — Kysely's `db.transaction()` reconnects libsql and drops FTS5 virtual tables on `:memory:`). The user-facing confirmation lines are now emitted only after a successful `COMMIT`, so a rolled-back reset is never partially announced. If `ROLLBACK` itself fails, the original error is still rethrown and a `!! ROLLBACK failed …` line is written to stderr so operators know the DB may need a manual check. Implementation: `buildResetCommand` in `src/cli/commands/memory.ts`.
- **`council expert train --retrain` honest rollback messaging (#383, #425)** — `DocumentRepository.clearForRetrain` now throws a typed `ClearForRetrainError` whose `rollbackFailed: boolean` field records whether `ROLLBACK` itself succeeded, alongside the original `cause` and any `rollbackError`. The `BEGIN` statement is also wrapped so a connection-level failure before any mutation still surfaces through the same typed contract (with `rollbackFailed: false`). The CLI now defaults conservatively: it only emits "Existing profile and tracking preserved" when it receives a `ClearForRetrainError` with `rollbackFailed === false`; any other error type — or a rollback that itself failed — produces a "database may be in an inconsistent state … inspect the document_index and expert_documents tables before retrying" message instead. Previously the CLI swallowed `ROLLBACK` failures and unconditionally claimed preservation, which could falsely reassure users that the DB was intact when in fact the FTS index and tracked documents may have disagreed.
- **Persona profile analyzer: existingProfile fields injected unsanitized (#358, #366)**— `analyzeDocuments()` now runs all `existingProfile` fields (`communicationStyle`, `decisionPatterns`, `biases`, `vocabulary`, `epistemicStance`) through the shared `sanitizePromptField` (extracted to `src/core/prompt-sanitize.ts` and reused by `src/core/prompt-builder.ts`) before interpolating them into the analyzer meta-prompt. C0 control bytes are stripped, Unicode line breaks (NEL / U+2028 / U+2029) and CR/LF runs are collapsed to a single space, bracketed numeric section markers (`[N]`) are defanged to `(sec-N)`, and each field is length-capped at 2000 characters. The fence-specific `<` → `&lt;` escape is layered on top so a persisted field still cannot prematurely close the `<documents>` data fence. Adversarial regression tests cover fence-break document content, `[N]`-marker spoofing, runaway-length fields, and C0/Unicode-line-break payloads.
- **`DebatePersister` single-use enforcement (#120)** — calling `persist()` a second time on the same instance now throws with a clear "single-use" message instead of silently overwriting `#debateId` and leaking `#pendingTurnPosition` state across debates. Construct a fresh persister per debate.
